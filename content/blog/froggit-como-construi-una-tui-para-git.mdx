Froggit empezó un fin de semana. No tenía grandes expectativas — quería una forma más cómoda de usar Git desde la terminal sin salir de mi flujo de trabajo en Neovim.

Hoy tiene más de **230 estrellas en GitHub** y gente de varios países lo usa a diario. Acá cuento cómo llegué hasta ahí.

## El problema que quería resolver

Los clientes GUI de Git son potentes pero pesados. La línea de comandos pura es flexible pero requiere recordar demasiado. Quería algo en el medio: navegación por teclado, visualización del estado del repo, y operaciones comunes sin salir de la terminal.

Herramientas como `lazygit` y `tig` ya existían, pero o eran demasiado complejas o no encajaban con lo que yo buscaba.

## Las decisiones técnicas

Usé **Go** por una razón simple: produce binarios estáticos que puedes distribuir sin dependencias. El usuario descarga un archivo, lo ejecuta, y listo.

Para la interfaz usé [Bubble Tea](https://github.com/charmbracelet/bubbletea), un framework TUI que sigue el patrón Elm. La arquitectura es limpia: tienes un modelo, un mensaje, y una función de actualización. Muy predecible, fácil de testear.

```go
type Model struct {
    files    []GitFile
    cursor   int
    selected map[int]bool
}
```

El estado de la UI vive en un struct simple. Cada acción del teclado genera un mensaje que modifica ese struct. Sin magia.

## Lo que me costó más

Integrar comandos Git de forma confiable fue lo más complejo. No hay una librería oficial de Git para Go que sea completa, así que terminé ejecutando `git` como subproceso y parseando el output.

El problema: el formato del output de Git cambia entre versiones. Tuve que escribir parsers bastante robustos y manejar edge cases raros (repos sin commits, estados de rebase, merge conflicts).

## Por qué llegó a 230 estrellas

No hice campaña de marketing. Publiqué en Reddit `/r/commandline` con un GIF mostrando el flujo básico. Eso fue todo.

Lo que funcionó: **el GIF mostraba el valor en 10 segundos**. Sin texto, sin explicaciones. La gente lo vio y entendió por qué querría usarlo.

Lección: cuando construyes herramientas para desarrolladores, el mejor pitch es verla funcionar.

## Qué viene

Estoy trabajando en soporte para worktrees de Git y una integración más profunda con GitHub CLI. Si te interesa contribuir o solo mirar el código, está en [github.com/thewizardshell/froggit](https://github.com/thewizardshell/froggit).
